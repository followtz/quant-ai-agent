## 二、关键发现

### 🔴 Cron 任务 Token 消耗超预期（最重大发现）
- **问题**：原本设计 30 秒超时的轻量 Cron，实际消耗 29K–53K/次
- **实测数据**：22日 07:00–09:34（2.5小时）共运行 8 次，累计消耗 ~350K
- **推算**：Cron 任务日均消耗 **3,000K–5,000K**，占总预算 **7.5%–12.5%**
- **优化方向**：
  - 扩大 Cron 超时限制（现有 30 秒不够，改为 60-120 秒）
  - Cron 任务执行频率可适当降低（如 15分钟→20分钟）
  - Cron 任务应避免调用大模型，只做 HTTP 检查 + 状态写入

### 🟠 Heartbeat Token 消耗波动极大
- 最低：~19K（纯检查，无大模型调用）
- 最高：~193K（带富途桥接 + dashboard写入 + 大模型处理）
- 建议 Heartbeat **严禁调用大模型**，严格控制在 30K 以内

### 🟡 LCM 上下文压缩消耗高
- Internal Context 注入：21K–709K 不等
- 大上下文窗口时压缩成本高，建议控制对话长度

### 🟢 对话总结稳定可控
- 每次：13K–21K，平均 ~16K
- 属于正常开销，无需特别处理

## 四、优化建议（已同步更新）

| 编号 | 建议 | 影响 |
|------|------|------|
| T-001 | Cron 任务改为 60-120 秒超时（不要用 30 秒） | 避免 SIGKILL 杀死进程 |
| T-002 | Cron 任务执行后**不推送**（只写 JSON 快照） | 减少大模型调用 |
| T-003 | Heartbeat 严格控制在 30K 以内 | 避免高消耗 Heartbeat |
| T-004 | 每日 20:00 检查 Token 余量，提前预警 | 避免午夜耗尽 |
| T-005 | Cron 任务降低频率（15分钟→20分钟） | 节省约 20% Cron 消耗 |
