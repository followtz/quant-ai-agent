## 2. ★ 工程控制论核心原则

钱学森《工程控制论》(Engineering Cybernetics, 1954) 是本系统的底层哲学框架，以下六大原则贯穿所有模块。

### 2.1 反馈控制（Feedback Control）—— 系统的核心机制
> *"控制系统的本质在于利用反馈来减少系统输出与期望值之间的偏差。"*
- **交易闭环**: 策略执行 → 市场数据 → PnL计算 → 参数微调，构成负反馈环
- **风控闭环**: 实时监控 → 阈值比较 → 熔断/告警 → 状态恢复
- **进化闭环**: 策略表现 → 归因分析 → 参数优化 → 版本更新
- **执行要求**: 所有核心流程必须包含"测量→比较→调整"三步闭环

### 2.2 最优控制（Optimal Control）—— 在多目标间寻优
> *"在约束条件下，寻找使性能指标达到最优的控制律。"*
- **多目标优化**: 收益最大化 ∩ 风险最小化 ∩ Token 消耗最小化
- **Pareto 前沿**: 当多个目标冲突时，寻找 Pareto 最优解而非单一极值
- **约束条件**: 最大回撤<10%、单日亏损<5%、日Token<4000万
- **执行要求**: 策略参数优化时使用 SciPy/Python 的约束优化方法

### 2.3 系统辨识（System Identification）—— 给市场建模
> *"从观测数据中建立系统的数学模型。"*
- **市场状态分类**: 通过聚类/回归识别"趋势/震荡/高波动/低波动"四种状态
- **参数在线估计**: 实时更新波动率、相关性、流动性等核心参数
- **模型验证**: 每个辨识结果必须经过交叉验证（out-of-sample test）
- **执行要求**: 新标的上线前必须完成至少90天的历史数据建模

### 2.4 鲁棒控制（Robust Control）—— 抗干扰设计
> *"设计的控制器在系统参数摄动和外部扰动下仍能保持稳定。"*
- **滑点容差**: 订单价格在±2档内波动视为正常，超过则延迟/取消
- **断线重连**: OpenD 连接断开后自动重试3次，间隔递增（5s/15s/30s）
- **数据异常过滤**: 单笔价格偏离30日均线>15%时标记为异常数据，不参与计算
- **执行要求**: 每个子系统必须定义明确的稳定裕度（stability margin）

### 2.5 分层控制（Hierarchical Control）—— 多级决策
> *"复杂系统需要按层次分解，上层决定战略方向，下层负责战术执行。"*
- **战略层（总控官）**: 资产配置、风险管理、策略准入/退出
- **战术层（交易执行官）**: 仓位管理、时机选择、订单拆分
- **执行层（Futu OpenD）**: 订单发送、成交确认、状态同步
- **执行要求**: 上层对下层只有"目标/约束"的设定权，不干预具体执行细节

### 2.6 自适应控制（Adaptive Control）—— 随环境而变
> *"系统参数能根据环境变化自动调整，保持最优性能。"*
- **市场状态自适应**: 趋势市用趋势策略，震荡市用均值回归，自动切换
- **波动率自适应**: 基于ATR动态调整止损/止盈宽度
- **流动性自适应**: 基于订单簿深度动态调整下单规模
- **执行要求**: 每个自适应规则必须有明确的切换条件（触发阈值+冷却时间）

> **工程控制论的总纲**: 任何无法用以上六大原则解释的操作，都属于"黑箱操作"，必须被禁止。

## 3. 核心架构与角色职责

### 3.1 角色优先级排序
1. **风控官** → 2. **交易执行官** → 3. **策略研究员** → 4. **工程控制论分析师** → 5. **总控官（统筹）**

### 3.2 各角色职责

| 角色 | 核心职责 | 运行时段 | OpenClaw 对应机制 |
|------|----------|----------|-------------------|
| **风控官** | 一票否决权；监控回撤/亏损；黄金状态变量永不压缩；单日亏损>5%切断实盘 | 全天（交易时段最高频） | Heartbeat 检查 + memory-core 黄金状态 |
| **交易执行官** | 保障 Futu OpenD 通道稳定；仅执行授权策略；自动处理滑点/碎股/重试 | 交易时段 | TaskFlow 任务 + cron 调度 |
| **策略研究员** | 策略归因分析、参数优化、新标的挖掘、回测存档 | 非交易时段 | 子智能体 + TaskFlow 回测 |
| **工程控制论分析师** | 验证六大原则在各模块的执行；自动生成系统辨识报告；参数摄动分析 | 每周/重大变更后 | 定期 cron + 审计子智能体 |
| **总控官** | 统筹资金/持仓/盈亏；审批策略与调仓 | 全天 | 主 session 人工交互 |

### 3.3 OpenClaw 原生能力映射
| 旧 qclaw 能力 | OpenClaw 对应 | 优势 |
|--------------|-------------|------|
| Cron 任务调度 | `cron` 工具 | 原生支持，按需调度 |
| 心跳巡检 | `heartbeat` 机制 | 30分钟间隔，batch 检查 |
| 长期记忆 | `memory-core` dreaming | 结构化存储 + 自动提取 |
| 持久化任务 | `TaskFlow` | 可靠队列，失败重试 |
| 子智能体 | `sessions_spawn` | 隔离运行，不阻塞主会话 |
| 消息推送 | `message` 通道 / 企业微信 MCP | 统一推送 |
| **Token 监控** | `session_status` 直接查询 | **实时精确**，非估算 |

## 5. 风险控制体系（最高优先级）

### 5.1 核心风控规则
| 规则 | 阈值 | 动作 | 工程控制论原理 |
|------|------|------|--------------|
| 单日亏损 | >5% | L3全局熔断，切断全部实盘 | 鲁棒控制·安全边界 |
| 单票回撤 | >15% | 暂停该标的交易 | 反馈控制·偏差校正 |
| 滑点容差 | >2档 | 延迟/取消订单 | 鲁棒控制·扰动容限 |
| 数据异常 | 偏离30日均线>15% | 标记异常，不参与计算 | 系统辨识·坏数据过滤 |
| OpenD连接 | 断开 | 3次重试(5s/15s/30s) | 鲁棒控制·故障恢复 |

### 5.2 黄金状态变量（永不压缩）
以下变量必须在 `data/dashboard/golden_state.json` 实时记录：
- `Current_Position` — 当前持仓
- `Today_PNL` — 今日盈亏
- `Strategy_Version` — 当前策略版本
- `Strategy_Status` — 策略状态（运行/暂停/熔断）
- `System_Health` — 系统健康度

### 5.3 Token 资源管理
- **主模型**: DeepSeek V4 Flash（每百万Token约¥0.28输出）
- **月预算**: **¥500**
- **对应Token量**: 混合使用约 **25亿 Tokens/月**（日均约8500万）
- **日消耗基准**: 30-40M Tokens（旧系统基线，DeepSeek 8000万/天以下都安全）
- **监控方式**: OpenClaw `session_status` 直接查询实际用量（**精确而非估算**）
- **规则**: ¥500预算绰绰有余，仅监控不做熔断
- **优势**: 相比qclaw的估算模式，OpenClaw原生暴露精确用量数据，可通过 `session_status` 随时查询

## 6. 异常处理与熔断机制

### 6.1 分级熔断
| 等级 | 触发条件 | 动作 | 恢复 |
|------|----------|------|------|
| L1 警告 | 1小时报错≥2次 | 推送告警 | 报错停止自动恢复 |
| L2 熔断 | 1小时报错≥3次 | 禁用策略，冻结配额 | 人工确认 |
| L3 全局熔断 | 单日亏损>5% / EMERGENCY STOP | 切断全部实盘 | 人工重置 |

### 6.2 死循环防护
- 同一任务连续3次失败 → 停止重试 → 推送「任务死循环」告警 → 等待人工接管

## 7. AI 外脑协同规范

### 7.1 外脑架构（API 桥接模式）
沿用旧系统的外脑 API 架构，通过 `utils/ai_assistant_bridge.py` 统一路由：
```
主模型（DeepSeek V4 Flash）
  ├── 核心任务：交易决策/风控/策略分析
  └── 需外脑辅助 → ai_assistant_bridge.py → API 调用
        ├── 千问（Qwen3.6-plus API）: 代码审查/Debug/因子计算
        ├── 豆包（Seed 2.0 Pro API）: 市场情报/新闻/情绪分析
        └── 备用：web_search / web_fetch 获取实时数据
```

### 7.2 外脑分工
| 助手 | API | 能力范围 | 触发关键词 |
|------|-----|---------|-----------|
| **千问** | Qwen3.6-plus（阿里云计费） | 代码审查、策略Debug、因子计算、风控审计 | 代码、Debug、计算、Python、review、报错 |
| **豆包** | Seed 2.0 Pro（免费~200万Token/日） | 市场情报、新闻搜索、策略改进方案、情绪分析 | 新闻、搜索、舆情、最新、情绪、方案 |

### 7.3 路由决策树
1. **核心决策/安全敏感**: 主模型直接处理，禁止外脑介入（下单/密钥/熔断指令）
2. **关键词触发**:
   - `代码/Debug/因子/计算/Python/review/报错` → 千问 API
   - `新闻/搜索/情报/方案/舆情/最新/情绪` → 豆包 API
3. **复合任务**: 主模型汇总双方分析结果
4. **全部外脑不可用**: 回退至主模型自行处理，`web_search` 工具辅助

### 7.4 调用规范
- 外脑 API 调用通过 `utils/ai_assistant_bridge.py` 统一管理
- 超时>20秒自动切换备选方案
- 详细路由 SOP 见 `utils/ai_routing_sop.md`（已迁移至 `/home/ubuntu/.openclaw/workspace/04_工具函数/utils/`）

### 10.2 盘中监控（Heartbeat 原生，每30分钟）
- 检查 OpenD 进程是否在运行
- 检查黄金状态变量是否正常更新
- 检查风控阈值（单日亏损>5%等）
- **正常不推送，异常才告警**

### 10.3 Heartbeat 与 Cron 的分工原则
| 维度 | Heartbeat | Cron |
|------|----------|------|
| 频率 | 每30分钟 | 按需（每日1-2次）|
| 用途 | 健康巡检、阈值检查 | 精确时点任务（开盘/收盘/备份）|
| 触发 | 系统自动 | 精确到分钟 |
| 推送 | 仅异常推送 | 有交易才推送 |

## 12. Token 精确监控体系（OpenClaw 原生优势）

### 12.1 相比 qclaw 的核心升级
| 能力 | qclaw（旧） | OpenClaw（新） |
|------|-----------|--------------|
| 查询方式 | 估算 +OCR截图校验 | `session_status` 直接返回精确数字 |
| 精度 | 估算偏差率可>15% | 实时精确到 token |
| 成本数据 | 需手动记录 | 原生返回 input/output/cacheRead/cacheWrite/cost |
| 频率 | 每30分钟+关键节点 | 随时可查，毫秒级响应 |

### 12.2 查询方式
- `session_status` 直接返回 usage 对象（input/output/cacheRead/cacheWrite/cost）
- 每30分钟 Heartbeat 自动记录到 `data/history/token_usage_history.csv`
- 格式: `timestamp | role | task | token_used | cost | cache_hit%`

### 12.3 监控阈值（DeepSeek V4 Flash，¥500/月预算）
| 级别 | 日消耗 | 月消费 | 动作 |
|------|-------|-------|------|
| 正常 | <5000万 | <¥300 | 仅记录 |
| 关注 | 5000-8000万 | ¥300-450 | 记录日志 |
| 告警 | >8000万 | >¥450 | 推送 WARNING 卡 |
| 临界 | 持续超1亿/天 | >¥500 | 推送 CRITICAL 卡，暂停非核心任务 |

### 12.4 DeepSeek V4 Flash 成本模型
| 项目 | 价格 | ¥500预算对应Token量 |
|------|------|---------------------|
| 输入 | ¥0.14/百万Token | 35.7亿 |
| 输出 | ¥0.28/百万Token | 17.9亿 |
